# 🔄 Поток на заявките (Request Lifecycle)

Този документ обяснява как данните преминават през системата, когато се извика метод в `DocumentController`.

---

## 🌊 Диаграма на процеса "Одобрение"

1.  **Request:** Потребителят изпраща заявка за одобрение на версия #5.
2.  **Fetch:** Контролерът извиква `versionRepository.findById(5)`.
3.  **Validate:** Ако версията не съществува, хвърляме `RuntimeException` (тук се намесва QA за Exception Handling).
4.  **Model Building:** Данните от базата се пакетират в бизнес обекти:
    - Създава се обект `Document`.
    - Създава се обект `Version`.
    - Дефинира се обект `User` с роля `REVIEWER`.
5.  **Service Call:** Всички тези обекти се подават на `workflowService.approveDocument`.
6.  **Response:** Контролерът връща текстово съобщение за успех към потребителя.

---

## 🛠️ Използвани технологии в API слоя

- **Spring Dependency Injection:** Използваме `@Autowired`, за да получим достъп до услугите и репозиторитата, без да ги създаваме ръчно.
- **Collections (Set/List):** Използваме `HashSet` за ролите, за да избегнем дублиране на права, и `List` за историята на версиите.
- **Error Handling:** Използваме `.orElseThrow()` за елегантно прекъсване на изпълнението при липсващи данни.

---

## 🎯 Цел на дизайна
Този дизайн позволява на **API Developer-а** да контролира кой има достъп до системата и какви данни се подават на бекенда, като същевременно пази базата данни защитена от директни външни манипулации.
